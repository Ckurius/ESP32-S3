Pinout & Control

Which pins you may use

Filter the GPIOs by function. Red = off-limits (occupied by flash/octal PSRAM) — never wire these pins, or the cache will crash. GPIOs 22–25 do not exist on the S3.

Free Off-limits (flash/PSRAM) Strapping (boot) Special (USB/UART/LED)
📌

Rules of thumb for practice

Off-limits: GPIO 26–32 (SPI flash) and 33–37 (octal PSRAM) — even though broken out: hands off.

Careful at boot: 0, 3, 45, 46 are strapping pins.

Occupied when using USB: 19 (D−) / 20 (D+).

Free & hassle-free: 1–18, 21, 38–42, 47.

# Drive the RGB LED on GPIO48
import neopixel, machine
np = neopixel.NeoPixel(machine.Pin(48), 1)
np[0] = (60, 0, 90)   # RGB
np.write()

# Set a free pin as output
p = machine.Pin(5, machine.Pin.OUT)
p.value(1)
Usability & Quick Start

Ready in 5 steps

From unboxing to the first glowing LED — including the typical Linux hurdles.

Plug in USB‑C “COM”

The port next to the CH343 bridge (labeled “COM”). It delivers power and data — one port is enough.

Find the device

Shows up as /dev/ttyACM0 (not ttyUSB! The CH343 uses cdc_acm).

Set permissions

Once: sudo usermod -aG dialout $USER + log in again. Instant fix: sudo chmod a+rw /dev/ttyACM0.

Flash MicroPython

Important: use the SPIRAM_OCT variant — only it enables the 8 MB of octal PSRAM.

Connect & get going

mpremote connect /dev/ttyACM0 opens the REPL. Done.

🔋
Power supply: Both USB‑C sockets deliver 5 V. The “COM” port alone is enough for powering and flashing. You only need the native “USB” port for USB device classes, JTAG debugging, or maximum USB throughput.
⚡
Two USB ports — the difference: “COM” (CH343) has auto-reset for flashing and works with every tool. The native “USB” comes straight out of the SoC (USB‑CDC/HID/JTAG) and is faster.
# Flash MicroPython (octal!)
esptool --chip esp32s3 -p /dev/ttyACM0 erase-flash
esptool --chip esp32s3 -p /dev/ttyACM0 -b 921600 \
  write-flash -z 0 ESP32_GENERIC_S3-SPIRAM_OCT-*.bin

# Open the REPL
mpremote connect /dev/ttyACM0

The most important quirk: Genuine octal PSRAM (8 MB) is the big advantage of this N16R8 — but it has two consequences: you must select OPI/SPIRAM_OCT when building, and GPIO 33–37 are permanently occupied. Many “R8” boards on the market actually carry quad PSRAM — on this board it has been confirmed as octal by a runtime test.

Original or clone? “N16R8” is Espressif's module designation (N16 = 16 MB flash, R8 = 8 MB octal PSRAM). According to the official datasheets, however, the genuine ESP32-S3-DevKitC-1 has Micro-USB (not USB-C) and is only listed as N8R8 or N32R16V — an off-the-shelf “DevKitC-1-N16R8” is not a standard variant. So your board is a third-party clone (often “YD-ESP32-S3”) with genuine Espressif silicon inside — recognizable by the CH343 bridge (original: CP2102N).

The key takeaway: For USB, GPIO, flash, WiFi, crypto, dual-core, the firmware was the bottleneck — with custom ESP‑IDF/C firmware the numbers explode (native USB ~65×, GPIO ~30×, flash ~10×; true 2-core parallelism instead of the GIL). 🔬 How it was measured — the measurement process

🐍 MicroPython 1.28

Interactively via the REPL (mpremote): on-device tests (PSRAM stress, touch, ADC, temp sensor) plus WiFi, USB, and BLE throughput. BLE counterpart: BlueZ via bleak + smartphone.

🏗️ ESP‑IDF v6.0.1 (custom C firmware)

On-device benchmarks at boot (UART). WiFi = ESP as access point + TCP server, PC connected directly. USB = USB-Serial-JTAG (native port). BLE = NimBLE notify, cross-checked with BlueZ and nRF Connect.

🎯
Measurement principle: always single-hop (no router/hub in between) and host and device side captured simultaneously — both had to match, otherwise invalid. 32–64 MB per throughput test. Firmware/scripts: idf‑throughput/ · idf‑ble/ · Tiefenpruefung/.
🚀 Maximum firmware idf‑max — the last limit pushed

A custom ESP‑IDF firmware (C, v6.0.1) with three modes tuned to the hardware limit, switchable via UART0 (USB / WiFi / BLE). It answers the question: Where was the firmware really the bottleneck — and where is the true hardware wall? The measurements appear above as the third (teal) bar in the comparison chart.

📡

WiFi — big leap

The bottleneck was TCP overhead, not the HW. UDP + AMPDU + PS_NONE + large buffers → ~50 Mbit/s (host-confirmed, 0 % loss). Close to the practical 1×1 802.11n HT20 limit.

🔌

USB — silicon wall

Even the theoretically fastest path — TinyUSB vendor bulk via libusb, no tty — lands at ~6.3 Mbit/s, identical to USB-Serial-JTAG. USB 2.0 full-speed is the true limit, not a firmware limit.

📶

BLE — central-capped

A notify pipeline (15 in flight, 2M PHY, DLE) lifts ESP‑IDF from 0.53 to 0.81 Mbit/s (+53 %). The PC central (BlueZ) won't allow more — it dictates the event length.

🧭
Verdict of the maximum firmware: “Firmware, not hardware” holds true where software layers were in the way (WiFi TCP, MicroPython GIL/UART mirror) — there, C/ESP‑IDF extracts multiples. With USB & BLE, however, you hit real, immovable limits (the USB FS peripheral and the external BLE central) that no firmware can overcome. Project & reproduction: idf‑max/.
✅
Verified before release: A built-in compute self-test (mode c) checks CPU, RAM, flash, GPIO, HW SHA256, and PSRAM on the board — result PASS (dual-core ×2.00 · SRAM 726 MB/s · PSRAM 21 MB/s · SHA256 95 MB/s · 8 MB PSRAM detected). Two values additionally optimized: GPIO toggle 7.94 → 39.7 MHz (dedicated-GPIO instruction instead of APB registers) and flash read 8.7 → 11.4 MB/s (QIO instead of DIO). 120 MHz PSRAM/flash was tested but rejected as not deploy-safe (model-dependent flash HPM). All three radio/USB modes also boot cleanly. Plus a long-term burn-in (mode s): both cores measured at 100% load + continuous self-verification of PSRAM, SRAM, flash CRC, SHA and heap — over the runtime 0 errors, 0 unexpected reboots, no heap leak (temp stable ~40 °C).

Tools & Methods

What was used for testing — step by step

Every tool from this analysis with installation instructions and practical notes — plus the methodology behind it, deliberately explained for beginners too.

🧭
For newcomers first: You plug in the board via USB‑C, and a tool on the PC “talks” to the chip over the serial interface (/dev/ttyACM0). This lets you upload firmware (flash) and run programs. Below you'll find exactly the tools that make this possible.

🧰 The tools

⚡

esptool & espefuse

Espressif's command-line tool for flashing, reading chip info/eFuses, and for the flash backup. We used it to identify the chip, pull the 16 MB backup, and install MicroPython.

# recommended (isolated installation):
pipx install esptool
# alternative:
python3 -m pip install --user esptool
⚠️
Watch out: Always set --chip esp32s3. The port is /dev/ttyACM0 (not ttyUSB!). Always pull a backup before overwriting.
🐍

MicroPython (Firmware)

A Python runtime that runs on the chip itself — perfect for interactive experimentation without compiling. We used it to test PSRAM, WiFi, BLE, touch, ADC, and deep sleep.

# 1) Download the octal-PSRAM variant from micropython.org:
#    ESP32_GENERIC_S3-SPIRAM_OCT-*.bin
# 2) flash it:
esptool --chip esp32s3 -p /dev/ttyACM0 erase-flash
esptool --chip esp32s3 -p /dev/ttyACM0 -b 921600 \
  write-flash -z 0 ESP32_GENERIC_S3-SPIRAM_OCT-*.bin
⚠️
Watch out: Be sure to use the SPIRAM_OCT variant — only it enables the 8 MB of octal PSRAM. The standard variant otherwise shows only ~300 KB of RAM.
💬

mpremote

The official companion tool for MicroPython: opens the interactive console (REPL), runs scripts, and copies files to the board. Our main tool for the live tests.

python3 -m pip install --user mpremote

# open the console:
mpremote connect /dev/ttyACM0
# run a script:
mpremote connect /dev/ttyACM0 run my_test.py
⚠️
Watch out: When connecting, mpremote performs a soft reset — which overwrites e.g. the deep-sleep wake cause. Use pyserial for such cases.
🔌

pyserial

Python library for direct serial access — no intermediate layer. We used it to read boot logs, trigger resets, and take raw throughput measurements.

python3 -m pip install --user pyserial

# Trigger a reset (RTS) + raw read — example:
import serial, time
s = serial.Serial('/dev/ttyACM0', 115200)
s.rts = True; time.sleep(.1); s.rts = False
⚠️
Watch out: DTR/RTS control the ESP's reset/boot. The baud rate (115200) must match the firmware — with native USB it doesn't matter.
🏗️

ESP‑IDF (C‑Toolchain)

Espressif's professional development framework (in C). Required to push the full hardware — in our case: true dual-core speedup and the maximum native-USB throughput.

mkdir -p ~/esp && cd ~/esp
git clone -b release/v5.3 --depth 1 \
  --recurse-submodules --shallow-submodules \
  https://github.com/espressif/esp-idf.git
cd esp-idf && ./install.sh esp32s3
. ./export.sh        # required in EVERY new shell
⚠️
Watch out: Several GB of download. Re-source . ./export.sh in every shell. Once per project: idf.py set-target esp32s3, then idf.py build flash.
📶

bleak

Cross-platform Python library for using the PC as a Bluetooth LE central. With it, the computer connected directly to the ESP and measured the BLE throughput.

# due to PEP 668, into a virtual environment:
python3 -m venv ~/bleenv
~/bleenv/bin/pip install bleak
~/bleenv/bin/python my_ble_test.py
⚠️
Watch out: Modern Linux distros block system-wide pip install (PEP 668) → use a venv or pipx. The Bluetooth service (bluetoothd) must be running.
🌐

NetworkManager / nmcli

Linux tool for WiFi control. We used it to connect the PC to the ESP access point for the clean single-hop WiFi measurement, or to spin up a PC hotspot.

# usually preinstalled. Connect to the ESP AP:
nmcli device wifi connect "ESP32-Speedtest" password "…"
# turn the PC itself into a hotspot:
nmcli device wifi hotspot ssid "PC-Hotspot" password "…"
⚠️
Watch out: When switching to the ESP WiFi, the PC loses its previous WiFi (Ethernet stays). Switch back afterwards with nmcli connection up <name>.
🐧

Linux basics (USB & serial)

Built-in tools to detect the board and get access: lsusb, udevadm, dmesg, plus the group permissions.

# see the board (VID:PID):
lsusb | grep -iE "303a|1a86"
# permanent access (once, then log in again):
sudo usermod -aG dialout $USER
# instant fix for the current session:
sudo chmod a+rw /dev/ttyACM0
⚠️
Watch out: The CH343 uses the cdc_acm driver → the device is called ttyACM, not ttyUSB. After every re-plug/mode switch the chmod may be needed again.

🔬 The methodology

Good measurements need the right method. These principles made our results reliable — each briefly explained with why you do it that way.

💾

Back up first, then modify

Before the first flash, pull the complete 16 MB flash as a backup. Why: The factory state can be restored at any time — experiment risk-free.

🔀

Octal vs. quad: test both builds

Datasheet codes can deceive. We flashed both MicroPython variants and saw from the heap which PSRAM is really installed. Why: Measuring beats guessing.

⚖️

Measure on both sides

Always capture throughput on the host and device side simultaneously. Why: If both values match, the result is reliable and not a measurement artifact.

🔁

Internal UART loopback

Route TX and RX of a free UART to the same pin → read data back without a cable. Why: This finds the chip's max baud rate without any extra hardware.

📡

Single-hop WiFi

For true WiFi throughput, run the ESP as its own access point and connect the PC directly. Why: No router/hotspot in between to distort the result (double-hop halves it!).

📜

Reset cause from the boot log

When waking from deep sleep, the ROM prints rst:0x5 (DSLEEP) on the UART — read it raw with pyserial. Why: mpremote's soft reset would otherwise overwrite the cause.

📦

Isolate Python cleanly

Install packages in a venv or via pipx, not system-wide. Why: Modern distros block that (PEP 668), and it keeps the system Python clean.

🔧

Bootloader mode for native USB

In the bootloader, the native USB enumerates as USB-Serial-JTAG (303a:1001) — esptool can then talk over it. Why: Access to the native port without custom firmware.

🎯

The right tool for the goal

MicroPython for quick exploration, ESP‑IDF when the hardware should hit its limit (dual-core, USB speed). Why: Many “limits” are firmware-, not hardware-related.

Extra ③ · Troubleshooting

Pitfalls & solutions

The mistakes almost everyone makes once with this board — and how to fix them immediately.

🧱

Only ~300 KB RAM instead of 8 MB

You flashed the non-octal firmware. Solution: use the SPIRAM_OCT variant — only it talks to the octal PSRAM.

🔍

No /dev/ttyUSB0

Correct — the CH343 enumerates as /dev/ttyACM0 via cdc_acm. Look for ttyACM*, not ttyUSB*.

🚫

“Permission denied”

Your user is not in dialout. sudo usermod -aG dialout $USER + log in again, or quickly sudo chmod a+rw /dev/ttyACM0.

💥

Crash on GPIO 33–37

These pins belong to the octal PSRAM. Any use crashes the cache. Use 1–18, 21, 38–42, 47 instead.

🔁

Which of the two ports?

For flashing, use the “COM” port (CH343, auto-reset). Shows up as 303a:1001/4001 → that's the native USB.

🐢

USB/serial is slow

The REPL runs at 115200 (~11 KB/s). For speed: native USB + ESP‑IDF (up to ~6 Mbit/s) or a higher baud rate in your own project.

🧬 Silicon errata (rev v0.2) & MicroPython configs

ℹ️
Good news first: Your v0.2 is the latest ESP32‑S3 revision — there is no “hardware update”, only the errata/ROM patches in ESP‑IDF (current with v6.0.1). Of 8 open errata, most are caught automatically below the app level; MicroPython inherits them. Only three affect your own usage — here's the solution:
📈

ADC‑183 · ADC2 + WiFi

With WiFi/BLE active, ADC2 (GPIO11–20) delivers distorted values. Solution: for analog reads use only ADC1 pins GPIO1–10 → ADC(Pin(1)).

✋

TOUCH‑100 · touch ISR

The touch interrupt raw value is undefined. Solution: poll touch instead of using an ISR → query TouchPad(Pin(x)).read() in a loop.

🌈

RMT‑176 · WS2812 continuous TX

In RMT continuous mode the idle level is wrong. Solution: NeoPixel in single-shot np.write() (MicroPython default) — no loop TX.

✅
Did the errata distort our measurements? No. We read ADC on GPIO1–4 = ADC1 (the right choice), touch via polling, WS2812 in single-shot, and deep instead of light sleep — all affected cases avoided. The throughput tests (USB/WiFi/BLE/GPIO/flash/PSRAM) touch none of the errata peripherals. The measured brakes were purely firmware-side (UART stdout mirror, GIL).
